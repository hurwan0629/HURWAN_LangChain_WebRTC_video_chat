# 2026-07-10 18:07:07 - 사용자의 인증 흐름 정리
> 해당 작업을 쓰는 이유는 사용자가 새로고침 및 네트워크 지연 등의 문제로 인해 튕기게 되면 매 페이지마다 다시 연결하는 과정이 필요할 수 있겠다는 생각이 들기도 하고 `accessToken`과 `refreshToken`을 잘 활용하려면 매번 `tokenExpired` 이벤트 등에 대해서 또 다룰 필요가 있기도 하고, `express.Router`의 미들웨어는 둘째쳐도 socket.io의 미들웨어 (auth 계층) 을 한번 알아봐야하기 때문에 이 작업을 한번 하기로 하였습니다.


만들것들을 대략적으로 생각해보면
- 로그인 상태 확인 API 
- refresh토큰은 /auth/refresh에서만 허용
- (정보) socket.io는 핸드셰이크(최초)에 쿠키 사용

## 로그인 상태 확인 절차 (모듈)
#### `index.html`
`/me` 
- 200이면 dashboard
- 401이면 `/refresh` 이후 다시시도

### `dashboard.html`
`/me`
- 200 이면 정상 표시 -> socket 연결
- 401이면 `/refresh`
  - 성공 시 위 작업 실행
  - 실패 시 `/index.html?error=session_expired`

### `p2p.html`/`room.html`
`/me`
- `socket` 연결 -> 방 연결 로직
- 실패하면 `/refresh` 
  - 성공하면 위의 작업
  - 실패하면 `index gogo`
